# CV 計測の検証チェックリストと本番反映フロー

実装が終わってからが本番。**単発で値が合っていても「正常」とは言えない**のが CV 計測の世界。以下のチェックを全て通してから完了報告する。

## 検証環境の鉄則

| 鉄則 | 理由 |
|------|------|
| **必ず Playwright MCP を使う**（playwright-cli 禁止） | playwright-cli は Headless 検出され、adebis 等の商用アドトラッキングがリクエストをブロックする（送信自体が観測できなくなり、検証が成立しない） |
| **検証は新しいタブで実施** | sessionStorage の発火フラグ（例: `ebis_register_cv_fired`）が前回検証の残骸として残っていると CV が飛ばず、「実装が壊れている」と誤判定する |
| **単発観察での「正常」判定は厳禁** | Shopify Pixel API は条件によって稀に shop currency を返すことがある。1 回だけ JPY 値が来たからといって「JPY で送信されている」と判断するのは間違い（CENE 案件で実際に誤判断し、後から訂正した経緯あり） |

## ピクセル送信値の検証: 最低 6 パターン

以下を**すべて**実施し、各パターンで送信値（金額・通貨・イベント種別）を記録する:

- [ ] ① クリーン状態（Cookie / storage なし）での初回
- [ ] ② 同商品の再追加
- [ ] ③ 別商品
- [ ] ④ 数量変更
- [ ] ⑤ カートクリア後
- [ ] ⑥ checkout_started まで進める

## 多通貨ストアの両側検証

多通貨ストア（Shopify Markets）では **TWD/JPY 両方の環境で**検証する。片側だけの確認は不可:

- [ ] 外貨側（例: TWD ロケール）: 換算ヘルパーが効いて **JPY 換算値**が送信されること
- [ ] JPY 側: 誤って為替レートが掛からない（**×1 素通しで二重換算されない**）こと

## 本番反映フロー

- [ ] **push 前に必ず最新の Live テーマ ID を確認する**:

  ```bash
  shopify theme list --store <store> | grep live
  ```

  memory・過去メモ・以前の作業ログにあるテーマ ID は古い可能性がある（CENE 案件 2026-06-03 に古い ID を信じて unpublished テーマへ push する事故が実際に発生）

- [ ] デプロイ後、ピクセルの読み込み URL の **`@N` バージョン番号がインクリメント**されたことを Network タブで確認（反映確認の指標。詳細は [pixel-currency-patterns.md](pixel-currency-patterns.md)）
- [ ] 本番環境でも上記 6 パターン（実施可能な範囲）+ 新規タブで再検証
- [ ] **検証で作成したテスト顧客の削除を依頼する**: テスト登録した顧客は本番 DB に残る。Shopify 管理画面 → 顧客管理からの手動削除を必ずユーザー（またはストア管理者）に依頼する

## 完了報告に含めるもの

- 6 パターンそれぞれの送信値の記録（金額・通貨・イベント）
- 多通貨ストアの場合は両通貨環境の結果
- 反映確認した `@N` バージョン番号
- テスト顧客の削除依頼を出した旨
